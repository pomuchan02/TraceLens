SELECT文からER図を自動生成する方法【JOIN解析】

はじめに

SQLのSELECT文からER図を作成する方法を解説します。

JOINが複雑でテーブル構造が分かりづらい場合でも、SELECT文だけで関係性を可視化できます。

ER図とは

ER図とは、データベースの構造を視覚的に表現した図です。

  • エンティティ（テーブル）
  • 属性（カラム）
  • 関係性（リレーション）

これらを図として表すことで、データの構造を直感的に理解できます。

SELECT文からER図を生成するメリット

構造の全体像を把握できる

結合テーブルが増えるほど、SQLだけでは構造理解が難しくなります。 ER図があることで、全体像を一目で把握できます。

保守・改修がしやすくなる

既存システムの改修時、ER図があると影響範囲の特定が容易になり 作業コストを削減できます。

テーブル定義不要

テーブル定義を探す手間なくても、目の前にあるSELECT文だけで構造を可視化できます。

誰でも理解しやすい

視覚化されることで、非エンジニアでも理解しやすくなり、新人のキャッチアップにも役立ちます。

なぜSELECTからER図が作れるのか

通常、ER図はCREATE TABLEなどのDDLから作成されることが多いですが、
実はSELECT文からでもテーブル同士の関係を読み取ることができます。

その理由は、SELECT文にはテーブル間の結合条件（リレーション）が明示されているためです。

例えば以下のSQLを見てみます。

SELECT *
FROM users u
JOIN orders o ON u.id = o.user_id;

このとき

u.id = o.user_id

という条件が記述されています。

これは

  • users テーブル の id カラム（主キー）
  • orders テーブル の user_id カラム（外部キー）

の関係を表しており、テーブル同士のリレーションそのものです。

つまり、JOIN + ON句 = ER図の線と考えることができます。

このように、SELECT文の中に含まれる結合条件を解析することで、ER図を自動的に生成することが可能になります。

JOINの仕組み

JOINは、複数のテーブルを結び付けてデータを取得するためのSQL構文です。
結合条件を指定することで、条件に一致するデータのみを結合します。

基本的な書き方は以下の通りです。

SELECT *
FROM テーブルA
JOIN テーブルB ON 条件;

この「条件」がテーブル同士の関係を定義しています。

代表的なJOINの種類

INNER JOIN - 内部結合
SELECT *
FROM users u
INNER JOIN orders o ON u.id = o.user_id;
  • 両方のテーブルに存在するデータのみ取得します。
  • 最も一般的なJOINです。
LEFT JOIN - 左外部結合
SELECT *
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;
  • 左のテーブル（users）は必ず残る
  • 右のテーブル（orders）は存在しない場合NULL
RIGHT JOIN - 右外部結合
SELECT *
FROM users u
RIGHT JOIN orders o ON u.id = o.user_id;
  • 右のテーブル（orders）は必ず残る
  • 左のテーブル（users）は存在しない場合NULL

ER図との関係

JOINは単なるデータ取得ではなく、どのテーブルとどのテーブルがどう繋がっているかを表現しています。

そのため、JOINを解析することで

👉 テーブル間の関係（ER図）を復元できる

というわけです。

ON句の見方（最重要）

ER図生成において最も重要なのがON句の解析です。
ON句には、テーブル同士の結びつきが直接書かれています。

基本パターン
u.id = o.user_id

この1行から読み取れることは以下です。

  • users.id と orders.user_id が関連している
  • usersが親テーブル
  • ordersが子テーブル
読み取り方のコツ
  • ① 「=」の左右を見る

    • 👉 それぞれ別テーブルのカラム

  • ② 「id」を持つ側が親になりやすい

    • users.id ← 親
    orders.user_id ← 子

  • ③ 複数JOINも同じ
    • JOIN order_items oi ON o.id = oi.order_id
    JOIN products p ON oi.product_id = p.id;

    👉 1つずつ分解すればOK

注意点
  • alias（u, o など）に注意する
  • 複数条件（AND）がある場合もある
  • ON o.id = s.order_id AND o.user_id = s.user_id
